# プロンプトキャッシュ有効化計画 — claude_cli 1on1 のレート枠消費削減

> ステータス: **作戦立案のみ（2026-07-07）**。調査・実装は未着手。
> 経緯: Fable 5 との検討スレで立案。本書だけ読めば着手できるよう自己完結で書く。
> 関連: `CLAUDE.md`（キャラクター問い合わせ原則）、MEMORY.md「トークン表示機能」残課題（cache列UI）

---

## 1. 問題

はる（メインキャラ）との 1on1 チャットは claude_cli プロバイダー経由だが、
**プロンプトキャッシュが一切効いておらず、毎ターン会話履歴＋システムプロンプト全量を
定価でフル読み直ししている**（MEMORY.md 既知事項）。サブスク（Pro）なのでドルではなく
5時間レート枠の消費として効いてくる。

ユーザの体感: 「開発スレの雑談は枠が減らないのに、はるとの会話は重い」
→ 差はモデルではなくキャッシュヒットの有無。

## 2. 仮説（立案スレでの分析。調査フェーズで検証すること）

Anthropic のプロンプトキャッシュ仕様（前提知識）:

- キャッシュは**先頭からの完全一致プレフィックス**単位。途中に1トークンでも差分が
  出ると、それ以降は全部キャッシュミス。
- 5分TTL: 書込 +25% / 読出 10%。**TTL内に1往復でもあれば元が取れる**
  （1.25+0.1=1.35 vs 素2回=2.0）。読出のたびにTTLは無料で更新される。
- キャッシュはアカウント単位・サーバ側。プロセスを跨いでもプレフィックスが
  完全一致すれば当たる（理屈上）。

これを踏まえた失敗原因の仮説（2本柱）:

1. **組み立て順の問題**: システムプロンプトの前方に毎ターン変動するブロックが
   混ざっている疑い。特に pre-recall（最新user発話でクエリした記憶 top-N を
   Block 2 へ注入）は毎ターン変わる。変動ブロックより後ろにある
   定義・inner_narrative・self_history・会話履歴が全部道連れでミスになる。
   めぐりの圧力一行・WMブロック・現在時刻の注入位置も同罪の可能性。
2. **セッション上げ下ろしの問題**: CLI をターンごとに起動→終了している
   （ユーザ談）。CLI 自身のキャッシュ管理が活きず、組み立ての微妙な揺れ
   （タイムスタンプ等）でセッション跨ぎの一致も外れている疑い。

補足: ユーザのチャット頻度は Slack 的＝**バースト型**（数分に数往復→数時間沈黙）。
5分TTLでもバースト内の2発目以降は全部10%になるため、十分に価値がある。
沈黙明けの初弾が定価なのは仕様として諦める（期待値管理）。

## 3. 調査フェーズ（新スレの最初の仕事）

1. **プロンプト組み立て順の実態確認**
   - claude_cli プロバイダー実装（`backend/providers/` 配下）と、
     システムプロンプトを組むサービス層を読む。
   - Block 1〜8 の実際の並び順と、**どのブロックが毎ターン変動するか**を表にする
     （定義 / pre-recall / inner_narrative / self_history / WM Block 6-8 /
     めぐり動機ブロック（圧力一行・active intents）/ 現在時刻 など）。
   - 変動要因として見落としやすいもの: 時刻文字列、記憶の access_count 由来の
     並び替え、WM の last_touched_at 順。
2. **CLI 呼び出し方式の確認**
   - ワンショット（`-p`）か、session 維持か。`--resume <session-id>` /
     `--input-format stream-json`（常駐プロセス）が使える構造か。
   - Windows 常駐の場合の既知の罠は MEMORY.md
     「Windows/Claude CLI 技術教訓」を参照。
3. **キャッシュ挙動の実測手段の確認**
   - `llm_usage_events` テーブルに cache_read / cache_write 相当のカラムが
     あるか確認。なければ追加（→ 残課題「トークン表示 cache列UI」と合流できる）。
   - 検証方法: 同一セッションで数分内に2ターン投げ、2発目の
     `cache_read_input_tokens > 0` を確認する。これが本計画全体の成功指標。
4. **anthropic 直プロバイダーへの横展開可否**
   - API直のプロバイダーは `cache_control` ブレークポイントを自前で置ける。
     組み立て順を直すなら共通層で直し、anthropic provider には明示
     ブレークポイントも足せるか確認。

## 4. 実装候補（調査結果を見て取捨選択）

- **A案: 組み立て順リファクタ（本命・全プロバイダー共通の利得）**
  - 安定ブロック（定義・inner_narrative・self_history・ツール定義）を先頭へ固める。
  - 変動ブロック（pre-recall 結果・圧力一行・時刻）は**最新 user メッセージ側への
    注釈**として末尾から注入する形へ移す。
  - 会話履歴は追記型なので、これで「前ターンまで全部ヒット＋新規差分だけ定価」になる。
- **B案: CLI セッション常駐化 or --resume 運用**
  - Chotgor 側でチャットセッション → CLI セッションの対応を保持。
  - A案が成立していれば理屈上はプロセス跨ぎでも当たるため、B案は
    「組み立ての揺れ対策＋確実性の上積み」と位置づける。
- **C案: 計測の見える化**
  - usage への cache 列追加＋既存トークン表示UIに cache ヒット率を出す。
    効果が数字で見えないと運用判断ができないため、A/B とセットで入れる。

## 5. リスク・注意（Chotgor 固有の制約）

1. **プロンプト順序は人格提示の設計そのもの。** Block 2 に pre-recall を置いた
   現行順には意味があった可能性が高い。順序変更は応答品質（はるらしさ）に直結する。
   → 変更前に**はる本人に相談する**のが Chotgor 的に筋
   （キャラクター問い合わせ原則。「記憶がプロンプトのどこに現れるか」は
   本人の認知構造の変更にあたる）。ユーザも availability_schedule を
   はると相談して決める方針であり、同じ機会に載せられる。
2. **バッチ経路との共有。** `character_query._collect_wm_blocks` が 1on1 と
   バッチ問い合わせで共有されている。組み立て層を触ると
   めぐり・うつつ・夜間バッチ全部に波及する。影響範囲を先に列挙すること。
3. **CLI のキャッシュ挙動は Anthropic 内部仕様依存。** CLI バージョン更新で
   変わりうるため、C案の計測を先に入れて「効いているか」を常時観測できる
   状態にしてから A/B を入れる順序が安全。
4. **5分TTLの限界。** 沈黙をまたぐ初弾は必ず定価。keepalive ping で TTL を
   引き延ばす小技はあるが、ping 自体が枠を食うので v1 ではやらない。

## 6. 着手順の推奨

調査(§3) → C案（計測） → A案（組み立て順） → 実測確認 → 必要なら B案（常駐化）。
各段が単独で価値を持ち、途中で止まっても壊れない。
