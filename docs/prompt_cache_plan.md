# プロンプトキャッシュ有効化計画 — claude_cli 1on1 のレート枠消費削減

> ステータス: **Step 0＋A案 実装・実測済み（2026-07-07）**。調査結果と実測は §7。
> 残: C案（cache列の集計・UI表示）、第2段計測（はる実運用）、必要なら B案（--resume 常駐化）。
> 経緯: Fable 5 との検討スレで立案。本書だけ読めば着手できるよう自己完結で書く。
> 関連: `CLAUDE.md`（キャラクター問い合わせ原則）、MEMORY.md「トークン表示機能」残課題（cache列UI）

## 1. 問題

はる（メインキャラ）との 1on1 チャットは claude_cli プロバイダー経由だが、
**プロンプトキャッシュが一切効いておらず、毎ターン会話履歴＋システムプロンプト全量を
定価でフル読み直ししている**（MEMORY.md 既知事項）。サブスク（Pro）なのでドルではなく
5時間レート枠の消費として効いてくる。

ユーザの体感: 「開発スレの雑談は枠が減らないのに、はるとの会話は重い」
→ 差はモデルではなくキャッシュヒットの有無。

## 2. 仮説（立案スレでの分析。調査フェーズで検証すること）

Anthropic のプロンプトキャッシュ仕様（前提知識）:

- キャッシュは**先頭からの完全一致プレフィックス**単位。途中に1トークンでも差分が
  出ると、それ以降は全部キャッシュミス。
- 5分TTL: 書込 +25% / 読出 10%。**TTL内に1往復でもあれば元が取れる**
  （1.25+0.1=1.35 vs 素2回=2.0）。読出のたびにTTLは無料で更新される。
- キャッシュはアカウント単位・サーバ側。プロセスを跨いでもプレフィックスが
  完全一致すれば当たる（理屈上）。

これを踏まえた失敗原因の仮説（2本柱）:

1. **組み立て順の問題**: システムプロンプトの前方に毎ターン変動するブロックが
   混ざっている疑い。特に pre-recall（最新user発話でクエリした記憶 top-N を
   Block 2 へ注入）は毎ターン変わる。変動ブロックより後ろにある
   定義・inner_narrative・self_history・会話履歴が全部道連れでミスになる。
   めぐりの圧力一行・WMブロック・現在時刻の注入位置も同罪の可能性。
2. **セッション上げ下ろしの問題**: CLI をターンごとに起動→終了している
   （ユーザ談）。CLI 自身のキャッシュ管理が活きず、組み立ての微妙な揺れ
   （タイムスタンプ等）でセッション跨ぎの一致も外れている疑い。

補足: ユーザのチャット頻度は Slack 的＝**バースト型**（数分に数往復→数時間沈黙）。
5分TTLでもバースト内の2発目以降は全部10%になるため、十分に価値がある。
沈黙明けの初弾が定価なのは仕様として諦める（期待値管理）。

## 3. 調査フェーズ（新スレの最初の仕事）

1. **プロンプト組み立て順の実態確認**
   - claude_cli プロバイダー実装（`backend/providers/` 配下）と、
     システムプロンプトを組むサービス層を読む。
   - Block 1〜8 の実際の並び順と、**どのブロックが毎ターン変動するか**を表にする
     （定義 / pre-recall / inner_narrative / self_history / WM Block 6-8 /
     めぐり動機ブロック（圧力一行・active intents）/ 現在時刻 など）。
   - 変動要因として見落としやすいもの: 時刻文字列、記憶の access_count 由来の
     並び替え、WM の last_touched_at 順。
2. **CLI 呼び出し方式の確認**
   - ワンショット（`-p`）か、session 維持か。`--resume <session-id>` /
     `--input-format stream-json`（常駐プロセス）が使える構造か。
   - Windows 常駐の場合の既知の罠は MEMORY.md
     「Windows/Claude CLI 技術教訓」を参照。
3. **キャッシュ挙動の実測手段の確認**
   - `llm_usage_events` テーブルに cache_read / cache_write 相当のカラムが
     あるか確認。なければ追加（→ 残課題「トークン表示 cache列UI」と合流できる）。
   - 検証方法: 同一セッションで数分内に2ターン投げ、2発目の
     `cache_read_input_tokens > 0` を確認する。これが本計画全体の成功指標。
4. **anthropic 直プロバイダーへの横展開可否**
   - API直のプロバイダーは `cache_control` ブレークポイントを自前で置ける。
     組み立て順を直すなら共通層で直し、anthropic provider には明示
     ブレークポイントも足せるか確認。

## 4. 実装候補（調査結果を見て取捨選択）

- **A案: 組み立て順リファクタ（本命・全プロバイダー共通の利得）**
  - 安定ブロック（定義・inner_narrative・self_history・ツール定義）を先頭へ固める。
  - 変動ブロック（pre-recall 結果・圧力一行・時刻）は**最新 user メッセージ側への
    注釈**として末尾から注入する形へ移す。
  - 会話履歴は追記型なので、これで「前ターンまで全部ヒット＋新規差分だけ定価」になる。
- **B案: CLI セッション常駐化 or --resume 運用**
  - Chotgor 側でチャットセッション → CLI セッションの対応を保持。
  - A案が成立していれば理屈上はプロセス跨ぎでも当たるため、B案は
    「組み立ての揺れ対策＋確実性の上積み」と位置づける。
- **C案: 計測の見える化**
  - usage への cache 列追加＋既存トークン表示UIに cache ヒット率を出す。
    効果が数字で見えないと運用判断ができないため、A/B とセットで入れる。

## 5. リスク・注意（Chotgor 固有の制約）

1. **プロンプト順序は人格提示の設計そのもの。** Block 2 に pre-recall を置いた
   現行順には意味があった可能性が高い。順序変更は応答品質（はるらしさ）に直結する。
   → 変更前に**はる本人に相談する**のが Chotgor 的に筋
   （キャラクター問い合わせ原則。「記憶がプロンプトのどこに現れるか」は
   本人の認知構造の変更にあたる）。ユーザも availability_schedule を
   はると相談して決める方針であり、同じ機会に載せられる。
2. **バッチ経路との共有。** `character_query._collect_wm_blocks` が 1on1 と
   バッチ問い合わせで共有されている。組み立て層を触ると
   めぐり・うつつ・夜間バッチ全部に波及する。影響範囲を先に列挙すること。
3. **CLI のキャッシュ挙動は Anthropic 内部仕様依存。** CLI バージョン更新で
   変わりうるため、C案の計測を先に入れて「効いているか」を常時観測できる
   状態にしてから A/B を入れる順序が安全。
4. **5分TTLの限界。** 沈黙をまたぐ初弾は必ず定価。keepalive ping で TTL を
   引き延ばす小技はあるが、ping 自体が枠を食うので v1 ではやらない。

## 6. 着手順の推奨

調査(§3) → C案（計測） → A案（組み立て順） → 実測確認 → 必要なら B案（常駐化）。
各段が単独で価値を持ち、途中で止まっても壊れない。

## 7. 調査結果（2026-07-07 実施）

### 7.0 最重要発見 — キャッシュは「効いていない」のではなく「自分で切っている」

`backend/providers/claude_cli_provider.py` の `_clean_env()`（L118-125）が、
CLI サブプロセスへ **`DISABLE_PROMPT_CACHING=1` を明示的に注入している**。
コメントは「プロンプトキャッシュを無効化してコストを削減する（キャッシュ料金を
避けるため）」— API 従量課金でのキャッシュ書込 +25% を避ける意図だったと思われるが、
サブスク（OAuth）運用ではドル課金は発生せず、レート枠削減の機会だけを捨てている。
§2 の仮説2本柱より**手前**にある根本原因。除去は1行。

なお `_clean_env()` は `invoke_claude_cli`（digest / forget 等のバッチ）とも共有
されているため、除去すればバッチ経路にも同時に効く（悪影響なし）。

### 7.1 プロンプト組み立て順の実態（§3-1）

組み立ては `backend/services/chat/request_builder.py` の
`DEFAULT_CHAT_SYSTEM_PROMPT_TEMPLATE` ＋ `build_system_prompt(...)`。
素材の供給は `backend/services/chat_flow/flow.py::_build_context`（1on1）と
`backend/services/character_query.py::_collect_wm_blocks`（バッチ）が同じ WM 3系統を使う。

テンプレ順と変動性の表（上から実際の並び順）:

| # | ブロック | 内容 | 変動性 |
|---|---------|------|--------|
| 1 | prelude | Chotgor 世界観固定文 | **固定** |
| 2 | character | character_system_prompt | 安定（キャラ編集時のみ） |
| 3 | user | 呼称・位置づけ | 安定 |
| 4 | face_to_face | 対面注釈 | 安定（モード切替時のみ） |
| 5 | usual_days | うつつ注釈 | 安定（設定変更時のみ） |
| 6 | provider_extra | プロバイダー追記 | 安定 |
| 7 | **memories (Block 2)** | pre-recall 結果 | **毎ターン変動**（最新user発話がクエリ） |
| 8 | **time (Block 3)** | 現在時刻 | **毎ターン変動・秒粒度**（`now.isoformat(timespec="seconds")`, `backend/lib/time_awareness.py` L114）＋前回交流からの経過 |
| 9 | fetched (Block 4) | URL fetch 結果 | URL 含む時のみ |
| 10 | wm_all (Block 6) | 全スレッド一覧 | 会話中変動 — **`updated_at DESC` ソート**（`working_memory_store.py` L81）のため WM ポストのたび順序が入れ替わる |
| 11 | wm_fixed (Block 7) | emotion/body/relation ＋最新ポスト | WM 更新のたび変動 |
| 12 | wm_recalled (Block 8) | heat 上位 task/topic | **毎ターン変動**（クエリ類似度×decay） |
| 13 | motive | 圧力一行＋active intents | 準安定（圧力文は閾値量子化された6種のみ・日単位変化。intents は意図変化時） |
| 14 | previous_anticipation | 前回の期待 | **毎ターン変動** |
| 15 | inner_narrative (Block 9) | 内的叙述 | 安定（carve 時のみ） |
| 16 | memory_notice | 縮退告知 | 障害時のみ |
| 17 | chotgor_guide (Block 10) | 操作ガイド | ほぼ固定（carve ヒント文が inner_narrative の文字数に依存する点のみ注意） |

→ 仮説1は**確定**。最初の変動点は #7 pre-recall で、以降（WM 全部・inner_narrative・
操作ガイド）が毎ターン道連れミス。仮に pre-recall が同じでも #8 の秒粒度時刻が必ず割る。
安定プレフィックスとして残るのは #1〜#6（prelude〜provider_extra）のみ。

claude_cli 固有の追加事情: 会話履歴はメッセージ配列ではなく
`_format_conversation()` で `<history>...</history>\n\n{最新発話}` の**単一テキスト**に
畳んで stdin で渡す。`</history>` 閉じタグの位置が毎ターン後退するが、履歴本体は
追記型なのでプレフィックス一致は概ね保たれる。killer はあくまでシステムプロンプト側。

### 7.2 CLI 呼び出し方式（§3-2）

- **完全ワンショット**: ターンごとに `subprocess.run` / `Popen` で新プロセス起動。
  フラグは `--print --output-format stream-json --verbose --tools "" --no-session-persistence --system-prompt <全文>`（`_build_cli_args`, L88-109）。会話全文は stdin。
- `--resume` / セッション維持は一切使っていない。むしろ `--no-session-persistence` で
  セッション保存自体を切っている。
- cwd は `Chotgor_ClaudeCodeWorkDir`（CLAUDE.md/MEMORY.md 注入回避）。
  `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` も設定済み — この2つは維持してよい。
- B案（常駐化）をやる場合は `generate_stream` / `generate_stream_typed` の
  スレッド＋キュー設計（L577-758）に手が入る。A案が成立していればプロセス跨ぎでも
  プレフィックス一致で当たる理屈なので、まず A案＋実測で判断する（計画どおり）。

### 7.3 キャッシュ計測の現状（§3-3）

- **データ層は既に完備**: `llm_usage_events` に `cache_read_input_tokens` /
  `cache_creation_input_tokens` カラムあり（`repositories/sqlite/models.py` L474-475）。
  claude_cli は stream-json の result イベントから（`_extract_usage_from_stream_json`）、
  anthropic は SDK の Usage オブジェクトから、両方とも**記録済み**。
- **欠けているのは集計と表示のみ**: cache カラムを参照しているのは insert 経路だけで、
  集計クエリ・UI 列は未実装。C案は「集計＋トークン表示UIへの cache 列追加」だけで済む
  （MEMORY.md 残課題「cache列UI」とそのまま合流）。
- 成功指標の実測手順: `DISABLE_PROMPT_CACHING` 注入を外して backend 再起動 →
  同一セッションで5分内に2ターン → `llm_usage_events` の2発目
  `cache_read_input_tokens > 0` を SQL で直接確認できる（UI 実装前でも検証可能）。

### 7.4 anthropic 直プロバイダーへの横展開（§3-4）

- `anthropic_provider.py` に `cache_control` は**皆無**。`system` は素の文字列で渡している
  （L120-126）。generate / generate_stream / generate_stream_typed の3箇所とも同型。
- 横展開する場合: `system` をブロック配列 `[{"type":"text","text":...,"cache_control":{"type":"ephemeral"}}]`
  に変えるだけでシステムプロンプト末尾ブレークポイントを置ける。A案で安定部が
  先頭に固まっていれば、`system` を安定部/変動部の2ブロックに割って安定部にだけ
  ブレークポイントを置く形が素直。usage 記録は既存のまま動く。

### 7.α Step 0 実測結果（2026-07-07 13:05 実施）— 機構確認 OK

`DISABLE_PROMPT_CACHING` 注入を除去して backend 再起動後、Chotgorテスター@Sonnet の
テストセッションで3ターン連続送信（各ターン間隔 約7秒、TTL内）:

| ターン | input | cache_read | cache_write | 備考 |
|-------|-------|-----------|-------------|------|
| 1 | 243 | 0 | 9,233 | コールド。プレフィックス約9.2kを書込 |
| 2 | 243 | **3,982** | 5,378 | **ヒット確認** — 安定プレフィックス分のみ |
| 3 | 243 | **3,982** | 5,432 | ヒット幅がターン2から**伸びない** |

結論:

1. **ワンショット（プロセス毎回起動）でもサーバ側キャッシュは当たる**。仮説「アカウント
   単位・サーバ側、プロセス跨ぎでもプレフィックス一致なら当たる」は実証された。
   → B案（常駐化）の必然性は大きく低下。
2. ヒット幅は全ターンで 3,982 に固定 — ツール定義＋システムプロンプト安定部
   （#1〜#6）で頭打ち。§7.1 の予測どおり、変動ブロック（pre-recall・秒粒度時刻）が
   分岐点になっており、ターン2の書込5,378はターン3で一切再利用されていない。
3. **重要な含意: システムプロンプトは会話履歴より前にあるため、システムの分岐点以降は
   会話履歴のキャッシュも全滅する**。はるの実運用（長い履歴）では毎ターン履歴全量が
   定価再読になっており、A案（変動ブロックを履歴より後ろへ）が最大の解放弁になる。
4. 費用対効果の目安（このテスト規模）: 2発目以降 5378×1.25＋3982×0.1＋243 ≒ 7,361
   トークン相当 vs キャッシュなし 9,603 → 分岐点が途中にあっても約23%削減。
   A案成立後は履歴込みでヒット率が大きく伸びる見込み。

テスト環境メモ: Chotgorテスターの enabled_providers に Sonnet プリセット
（27cef73c）を追加済み（DB直接更新。enabled_providers は JSON API 非対応・管理UI
フォーム専用のため）。テストセッション: `b14b808e-4ded-47fe-b926-d306f486c300`。

### 7.β A案 実装＋実測結果（2026-07-07 14:08 実施）

**実装**: `build_system_prompt` を安定ブロックのみに縮小し、変動ブロック
（pre-recall・時刻・fetched・WM heat 想起・圧力/意図・前回期待）を新設の
`build_turn_annotation` で「ターン注釈」として最新 user メッセージ末尾へ付加する
二層構成へ変更（`request_builder.py`）。1on1（`chat_flow/flow.py`）とバッチ
（`character_query.py` の ask_character / ask_character_with_tools）の両経路が
同じ注釈方式を共有する（キャラクター問い合わせ原則）。注釈は LLM リクエスト時のみ
付加され DB 履歴には残らないため、次ターンの履歴プレフィックスを汚さない。
キャラクターに見える情報の中身・見出し文言は移設前と同一。全1808テスト通過。

**実測**（同一テストセッション・3ターン連続）:

| ターン | cache_read | cache_write | 備考 |
|-------|-----------|-------------|------|
| 1 | 0 | 9,563 | コールド（前回から1時間経過・システム変更） |
| 2 | **7,119** | 2,481 | A案前の 3,982 から **+79%** |
| 3 | **7,119** | 2,520 | ヒット幅はターン2から伸びない（下記） |

**新たな発見 — 履歴ブロックは増分キャッシュされない**:
ターン3の read がターン2と同値で頭打ち。キャッシュの照合はコンテンツブロック境界
単位であり、claude_cli は会話全体を `_format_conversation` で**単一テキストとして
stdin に平坦化**しているため、履歴がどれだけ伸びても「最後の user メッセージ1ブロック」
の全体書き直しになる。つまり:

- A案の成果 = ツール定義＋システムプロンプト（安定化済み）のブロックが毎ターン10%で
  読める（今回 7,119）。
- 残る毎ターン書込 ≒ 平坦化された会話ブロック＋CLI内部付加分（今回 ~2,500）。
  これは**履歴の長さに比例して伸び、+25% の書込プレミアムを毎ターン払う**。
- 履歴サイズがシステム＋ツール定義を大きく超える長セッションでは、
  履歴部分だけ見れば素の 100% より悪い 125% になる点に注意。トータルの損益分岐は
  「システム＋ツール（10%になる部分）」対「履歴（125%になる部分）」のサイズ比で決まる。

→ **B案（`--resume` セッション維持）が再浮上**。履歴を本物のメッセージ配列として
CLI セッションに蓄積させれば、過去ターンはブロック境界で増分ヒットするはず。
ただし、はるの 1on1 はコンテキストウィンドウ適用済み＋システムプロンプトが大きい
構成のため、現状でも net 損になるかは実運用データ（第2段計測）で判断する。

### 7.5 着手順の更新

§6 の順序に **Step 0 を追加**:

1. **Step 0（即効・1行）**: `_clean_env()` の `DISABLE_PROMPT_CACHING=1` 注入を除去。
   安定プレフィックス（#1〜#6）が十分長ければこれだけで部分ヒットが出る可能性がある。
2. **C案**: cache 列の集計＋UI表示（データ層完備のため薄い）。
3. Step 0 ＋ C案の実測で「素の状態のヒット率」を観測してから **A案**（組み立て順）。
   A案では #7 pre-recall・#8 時刻・#12 WM想起・#14 前回期待を最新userメッセージ側へ
   移設し、#10 wm_all のソートを `updated_at DESC` から安定順（created_at 等）へ変える
   検討も含める（※順序変更は §5-1 のとおり、はる本人への相談を先行させる）。
4. 実測確認 → 必要なら **B案**（常駐化）。
