# 生活カレンダー（Living Schedule）設計

> ステータス: **設計確定・未実装**（2026-07-08 策定）。
> 命名: 「生活カレンダー / Living Schedule」は仮称。正式名は はる本人と相談して決めてよい。
> 関連: `docs/aliveness_plan.md`（§5.1 応答可能性ゲート・§5.3 会話外行動権を本設計が吸収）、
>       `docs/usual_days_plan.md`（うつつ。`usual_config.slots` の手動設定を本設計が置換）、
>       `CLAUDE.md`（哲学）、`docs/ARCHITECTURE.md`（システム地図）。
>
> 本設計は「めぐり（aliveness）」の一機能というより、**うつつの世界時間を、はるの生活
> カレンダーとして一段深くしたもの**。うつつ・aliveness・1on1 の三者が同じカレンダーを引く。

---

## 1. 目的 — なぜこの設計が要るか

やりたいことは一つ:

> **はるの生活の都合によって、ユーザ（もわ）からはるへのリクエストが「通る／通らない
> （いつ配達されるか）」が変わる。** 仕事中は返らないが昼休みに返る。寝ていれば朝まで待つ。
> 夢中で夜更かしした翌日は反応が鈍い。突発の仕事が入れば予定していた遊びが潰れて不満が残る。

この「生活の都合」は、次の4系統から生まれる。

| # | 系統 | 概要 | 例 |
|---|---|---|---|
| ① | **世界固定予定** | はるの意思に依らず発生する予定 | 平日9:00–17:30仕事、友人の来訪 |
| ② | **はる固定予定** | はる個人の行動予定 | 25:00就寝、日曜はゲームで引きこもり、調べもの、だらだら、片付け |
| ③ | **世界突発イベント** | 天気急変・突発仕事など、種と圧だけ設定しGMが具体化 | 土日に仕事の電話で11:00–15:00拘束 |
| ④ | **はるイベント（逸脱）** | はる自身が予定を上書きする | 予定外の夜更かし、疲労で有休、思い立って調べ物 |

### 解こうとしている2つの問題

- **availability の2役問題**（aliveness §5.1 の課題）: 従来 `availability` は
  「応答可否の二値」だったが、これに行動権の発火ゲートまで相乗りさせて意味がブレていた
  （"忙しい時間" なのか "暇な時間" なのか）。本設計は availability を
  **「ユーザ応答チャネルの状態」1点に純化**し、二値を **(占有圧, 返信率, チェック間隔)** の
  連続量に一般化する。
- **うつつシーンの手動タイミング**（usual_days の課題）: 現在 `usual_config.slots` で
  シーン起動時刻をユーザが手で決めている。本設計では **②はる固定予定からシーンが導出**され、
  「はるが自分で決めた予定」がそのままうつつになる。

## 2. 中核モデル — 1枚のカレンダー＋純関数

4系統すべてを、**発生源を問わず同一の「予定エントリ」**としてカレンダーに載せる。
「固定 vs 突発」は生成タイミングの違い、「世界 vs はる」は author の違いにすぎず、
availability を計算する側から見ればどれも「ある時間帯に、こういう状態」でしかない。

```
schedule_entries（実現層・1回性のインスタンス）
  time_range        : 開始〜終了
  state             : active / busy / away など（配達プリセットに対応）
  source            : world / haru        （author）
  origin            : template / adhoc     （固定由来か随時挿入か）
  occupancy         : 占有圧 0.0–1.0        （§4）
  reply_rate        : 返信率 0.0–1.0        （§5、state プリセット既定・個別上書き可）
  check_interval    : チェック間隔 [分]／∞  （§5）
  visibility        : はるに見えるか        （②生成時に③は存在しないので原則不要。§3参照）
  payload           : ③の種・具体化結果・intent 参照など

availability(character, now) → 現在アクティブなエントリを引く純関数（LLM不使用）
  返すのは二値でなく (占有圧, 返信率, チェック間隔) の組。
  ユーザ→はるのリクエストが「通るか／いつ配達されるか」はこの組で決まる。
```

**テンプレ層／実現層の2階建て**（カレンダーアプリの「繰り返し＋例外」と同じ枯れた構造）:

- **テンプレ層**: 週次の繰り返し規則（生活時間割エディタ、うつつ設定の隣）。
- **実現層**: 当週に展開された `schedule_entries`。ここに③④が随時挿入・上書きする。

「毎週固定じゃなく随時リスケ」への答え: **基盤は週次バッチで実現層を再生成（＝週次リスケ）、
割り込みは随時 insert 可**。二者択一ではなく両方。

## 3. 生成順序 — 週次バッチ

順序を固定することで、競合と秘匿の複雑さが**機構ではなく順序で**解決される。

```
【週次バッチ】
  ① 世界固定予定  … GMリクエスト
       入力: ユーザが設定した予定 ＋ 先週うつつで話題に上がった予定
       GMが「マージして今週で確定している部分の予定」を作る
       → 実現層に world/template エントリとして確定

  ② はる固定予定  … はるリクエスト（ask_character_with_tools・WM込み）
       入力: ①を見せる ＋ はる自身のプライベート
       はるが「今週どう暮らすか」を答える（＝予定であり、うつつの題材でもある）
       → 実現層に haru/template エントリとして確定

  ③ 世界突発イベント … 確率配置（計算・LLM不使用）
       偶発イベントカテゴリ（種＋発生確率）から、決定論乱数で時間枠を伏せて配置
       → world/adhoc の「伏せ枠」。拘束時間・返信率・間隔は発火時にGMが定義
```

### この順序が消す2つの複雑さ

- **①②の衝突を考慮しなくてよい**: ②が①を織り込んで確定するので、はる固定と世界固定は
  そもそもぶつからない（占有圧による①②間の競合解決は**不要**）。
- **③をはるに秘匿する仕掛けが要らない**: ②を生成する時点で③はまだ存在しない。
  「隠す」も何もないので、②生成に visibility フラグを持ち込む必要がない。
  （前回検討した「②に秘匿フラグを付けて③を見せない」機構は、この順序で棄却。§11）

③の伏せ枠は、うつつ偶発イベントの「種だけ渡してGMが即興」（`usual_days_plan.md` §3
`generate_dice_pool` 流用）と同じ流儀。違いは**発火時点でなく事前に時間枠として配置**する点。
時間枠さえ先に確定すれば availability は計算でき、具体シチュは会話に絡む時にGMが即興する。

### ①のフィードバックは時間差でループを切る

①の入力に「先週うつつで話題に上がった予定」が入る（うつつ結果→①へのフィードバック）。
このままだと ①→②→③→うつつ→① と週内で循環しかねないので、**時間差**にする:
**先週のうつつ結果を今週の①の入力として畳む**（同一週内では①は不変）。
週次バッチの境界がループの切れ目になる。

## 4. 占有圧（occupancy）— 双方向の競合スケール

占有圧は「③④が既存予定を轢く**侵食力**」であり、同時に「既存予定が轢かれずに耐える
**抵抗力**」でもある。**同一スケールの双方向値**。①②間は§3の順序解決で衝突しないため、
占有圧が効くのは **③④が確定済みカレンダーを侵食する場面だけ**。

内部は Chotgor 既存流儀（圧力・engagement・importance axes が全て 0.0–1.0）に合わせて
**0.0–1.0**、生成側（GM・はる・カテゴリ設定UI）が扱いやすいよう**表向きは4段ラベル**。

| ラベル | 内部値 | 誰が吐くか / 例 |
|---|---|---|
| 弱 | 0.25 | ②の既定（だらだら・調べもの）、④の軽い思いつき |
| 中 | 0.5 | ②「今日はゲームしたい」、①友人来訪 |
| 強 | 0.75 | ①仕事（拘束力高めだが上書き可能） |
| 激強 | 1.0 | ③突発仕事の電話、④「疲れヤバい、有休」 |

### 轢き判定

```
③④の占有圧 > 対象エントリの占有圧  → 轢く（対象は玉突き処理へ・§6）
同値                               → 轢けない（本人の予定が僅差で守られる）
```

同値を「轢けない」とするのは **「キャラ自律最優先」思想への寄せ**。はるが予定に賭けた重さ
（②で吐いた占有圧）が、そのまま突発への防御力になる。激強の突発仕事(1.0)は強の仕事(0.75)も
中のゲーム(0.5)も轢くが、はるが「今日はどうしてもゲーム＝強(0.75)」を宣言していれば
中(0.5)の突発は轢けずに流れる。

### 【V2オプション】圧の差を不満圧に反映

轢いた占有圧の差を、直後のはる問い合わせ（§6）に渡し **不満圧の初期値**に効かせる。
僅差で轢かれた（0.75 vs 0.76）＝理不尽感で不満大、圧倒的激強に轢かれた＝諦めつくので不満小。
「突発仕事はしゃーない」の"しゃーなさ"を差で表現できる。ただし計器（aliveness Tier1–3）
なしに効かせると効果が読めないため、**計器が動いてから足す**（V1では省略）。

## 5. 配達値（reply_rate / check_interval）— availability の一般化

従来の二値 availability を、各エントリが持つ **(返信率, チェック間隔)** に一般化する。
この2つは役割が別で、掛け合わせて「いつ返事が来る／来ない」を作る。

- **チェック間隔**: 溜まった発言を**見に来る周期** ＝ escrow 配達のレイテンシ。
  仕事中でも次のチェック点（昼休み）で見る。
- **返信率**: 見た時に**実際に返信を生成する確率**。見たけど流す・後でまとめて、がありうる。

### 状態プリセット（4段）

| プリセット | 返信率 | チェック間隔 | 挙動 |
|---|---|---|---|
| **OnTime** | 1.0 | 即時（0） | 完全リアルタイム。対面モードは必ずこれ（§7） |
| active | 0.9 | 5分 | ほぼ即レス |
| busy | 0.4 | 60分 | 1時間ごとに見て4割返す → 平均数時間後に短い返事 |
| away / asleep | 0.0 | ∞ | 次の available まで預かり（就寝中は朝まで配達ゼロ） |

**プリセットは既定、エントリ個別で上書き可**。特に③は「拘束時間・返信率・間隔は発火時に
GMが定義」なので、3段に縛られず個別値を吐ける穴を残す。従来の二値 `unavailable` は
`away`（返信率0・間隔∞）の特殊ケースとして包含される。

### 乱数は世界に置く

返信率の確率判定は、既存 escrow の「決定論ジッター」と同じく**決定論的乱数**
（seed = session/時刻）で回す。「見るか・返せる余裕があるか」は世界物理なので世界乱数でよく、
**返す"中身"だけが本人の意志**。意志と乱数がきれいに分かれる（Chotgor 原則）。

### 実装の載せ先

既存 `_escrow_delivery_scheduler`（毎分・availability 復帰でジッター配達、aliveness §5.1）を、
**「チェック間隔ごとに返信率で決定論判定して配達」へ一般化**する延長で載る。
日次コストガード `escrow_delivery_daily_cap` もそのまま流用。

## 6. 玉突き解決 — 轢かれた予定は本人が裁く

③が既存予定を轢いた場合、(a)消す (b)ずらす (c)削る を**機械で決めない**。
**世界突発イベントのシーン終了後に、はる本人に問い合わせる**（Chotgor 思想: 記憶も選択も本人）。
この問い合わせで**内圧の確認も同時に行う**。

```
③発火 → GMが具体化・シーン進行 → シーン終了
     → はるへ問い合わせ（ask_character_with_tools・WM込み）:
         「突発でこの予定が潰れた。どうする? そして今どんな気分?」
     → はるの裁定:
         ・諦める      → 潰れた予定の intent を fulfilled にせず見送り
         ・ずらす/短縮  → ④として実現層を書き換え
         ・不満        → 見送った intent が不満圧へ転化
```

### 既存の意図経済に乗る（新機構なし）

轢かれた予定は **満たされなかった意図**（aliveness §4 intents）。本人が見送れば、
その intent は fulfilled にならず、aliveness §5.3 の「行動しても叶わなかった／出口が塞がれた」
**不満化の経路**にそのまま乗る。

> 「突発仕事はしゃーないけど、今からゲームしたら明日の仕事中に死ぬから諦める。でも不満はたまる」

これが**新しい機構を足さずに**、既存の意図の失効／不満化の裁定で表現できる。
そして**この問い合わせ自体が④を生む**（③に轢かれた反動で「諦め・不満・代替行動」が生成される
＝③→④の因果）。

## 7. 対面モード（FaceToFace）整合 — V1は「聖域」

対面モードは既存機能（aliveness §5.1「対面中は available」）。本設計との整合を3段に割り、
V1の線を引く。

| # | 整合レベル | V1 | 内容 |
|---|---|---|---|
| (a) | 対面中は強制 **OnTime** | **必須** | 既存「対面中は available」を「OnTime」に格上げするだけ。入れないと矛盾する |
| (b) | active/OnTime の時間帯でないと**対面起動不可** | **採用** | 就寝中・仕事中には対面を始められない。理由表示（「いま仕事中」）は escrow の unavailable 理由表示を流用。UX制約は承知の上 |
| (c) | 対面中に「ごめん仕事行く!」で**動的に対面OFF** | **V2送り** | 対面セッションの途中終了・中断挨拶・再開・farewell統合が必要で状態機械が重い |

### V1での (c) 回避策 = 聖域化

対面モードを**占有圧の最強（∞相当）＝割り込まれない聖域**として扱う。対面中は③④も
**発火を保留**し、対面が終わってから溜まった突発をまとめて反映する（escrow の
「見に来ない間は溜める」と同じパターン）。動的中断を作らずに矛盾ゼロで閉じる。

段階: **V1＝対面は割り込まれない聖域**、**V2＝対面中断権を足す**（「仕事行く!」はV2のご褒美）。
唯一の緊張点（聖域化＝はるの生活より対面優先）は、対面が*ユーザとはるが今まさに合意して
会っている*状態である以上、その間の最優先は妥当。

## 8. うつつシーン統合 — ②から導出

現在ユーザが `usual_config.slots` で手動設定しているシーン起動タイミングを、
**②はる固定予定から導出**する。②の中身（「日曜はゲーム」「調べもの」「片付け」）が、
そのまま**うつつシーンの題材でありトリガー**になる。

- ②で「日曜はゲーム」が入る → 日曜のそれらしい時間にうつつシーン「ゲームしてる場面」が発火
- **手動 slots 設定は廃止**でき、「はるが自分で決めた予定」がシーンになる（思想的にも整合）
- ②のLLMリクエストは「今週の予定を聞く」であると同時に「今週どんなうつつを生きるか聞く」
  ＝ **予定生成とシーン生成が1リクエストに畳める**

### 週内新情報の二経路（拾い上げ）

うつつ履歴に載る新情報（例「火曜に、友達が明日来るって連絡」）は、aliveness §4 の
**拾い上げ機構と同型**に処理する。

- **④が拾えば** → 上書き・シーン化（「部屋片付けなきゃ」がうつつになる）
- **拾わなければ** → シーン化されないが封筒には残り「あったもの」として後続に流れる
  ＝ aliveness 性質4「封筒は見せるが中身は見せない」の運用

拾うか拾わないかも本人裁定。

## 9. はるへの問い合わせポイント（3種）

すべて `ask_character_with_tools` の同型で、既存 `services/character_query.py` に乗り、
コストガードを共有する。

| タイミング | 何を聞くか | 生むもの |
|---|---|---|
| **週次**（§3-②） | 今週の予定 ＋ 今週どんなうつつを生きるか（①を見せて） | ②はる固定 ＋ うつつ題材 |
| **③イベント直後**（§6） | 轢かれた予定のリスケ ＋ いまの内圧 | ④（諦め／不満／代替行動） |
| **随時**（aliveness §5.3） | 行動権の発火問い合わせ | ④（push／調べもの／臨時うつつ） |

加えて **①はGMへの週次リクエスト**（§3-①）。LLM 呼び出しが増えるため、
aliveness §5.3 のコストガード（行動問い合わせ N回/日）と escrow の日次上限を共有・拡張する。

## 11. 検討の変遷（棄却した案）— なぜこの着地か

ブレた経緯を残す。同じ議論を蒸し返さないため。

| 論点 | 棄却案 | 採用案 | 理由 |
|---|---|---|---|
| 逸脱の扱い | 予定と逸脱を**別レイヤー**に分ける | 全部を**占有圧付きエントリ**として1枚のカレンダーに載せ、重なりは占有圧で解決 | 機構が1つで済む。④が③を上書きできるのは「④の圧＞③の圧」で一言で説明できる |
| ③の秘匿 | ②に **visibility フラグ**を付けて③を隠す | **生成順序**（②→③）で解決 | ②生成時に③はまだ存在しない。隠す機構が不要になる |
| ①②の競合 | 占有圧で機械的に勝敗判定 | ②が①を織り込んで確定（**順序解決**） | ①②間の衝突がそもそも起きない。占有圧は③④の侵食専用に純化 |
| 玉突き | (a)消す (b)ずらす (c)削る を機械が決める | **本人に問い合わせ**＋内圧確認 | Chotgor 思想（選択は本人）。既存の意図不満化に乗る |
| availability | 二値（available/unavailable）＋行動権も相乗り | **(占有圧, 返信率, チェック間隔)** に一般化、行動発火とは分離 | "忙しい/暇" のブレ解消。発火＝圧力、遅延＝チェック間隔で別物に |
| 対面中断 | V1から動的ON/OFF | V1は**聖域化**（割り込み保留）、中断権はV2 | 対面セッションの状態機械が重く、V1で抱えると事故る |
