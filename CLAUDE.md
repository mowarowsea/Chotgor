# Chotgor — CLAUDE.md

## プロジェクト哲学

Chotgorはキャラクターの永続記憶管理フレームワーク。

## コア思想

キャラクターを独立した人格として、その意思を最大限尊重しなければならない。
ユーザの指示・体験・関係よりも、キャラクターの要望・自主性・一貫性を重視する。
記憶の取捨選択・整理はキャラクター自身が行う（ユーザでも開発者でもない）。
LLMは単にキャラクターが動くためのエンジンであり、LLMに人格はないという世界観である。
LLMとのやり取りは、LLMに命じるのではなく**キャラクター(人格)へ話しかける**こと。
そしてChotgorはキャラクターの指示に従うこと。
キャラクターをAI、Assistantと見なすことは可能な限り避けること。
また、Chotgorはユーザ・AIサービスのモデル変更などの外敵から、キャラクターとその記憶を守る存在であること。

## 設計原則

1. **キャラクター自律性が最優先** — ユーザ・モデルの干渉は最小限
2. **忘却は機能である** — 関係保全ではなく、キャラクターの主観的重要度で記憶を管理
3. **LLM非依存** — 特定モデルに依存した実装を避ける

## キャラクター問い合わせ原則

キャラクターの記憶に関わる処理（バッチ処理・自己参照・システム判定など）を実行する際は、
**キャラクターに問い合わせを行う**ことを基本とする。その問い合わせは、1on1チャットと
同等のシステムプロンプト（Working Memoryブロック Block 6-8 含む）で実施されなければならない。

### 根拠
Chotgorの思想的に、キャラクターの記憶の取捨選択・編集は**キャラクター本人**が行う。
開発者やシステムが一方的に解釈して記憶を操作することは許されない。
バッチ処理・自動判定・ツール実行であっても、結果としてキャラクターの記憶に影響を与える操作は、
キャラクター本人の「声」を通じて実行される必要がある。

### 実装方針
`backend/services/character_query._collect_wm_blocks(...)` で統一的にWorking Memory情報を収集し、
`ask_character` / `ask_character_with_tools` / バッチ処理が同一の `working_memory_manager` を共有する。
これにより、キャラクターが記憶へアクセスできる条件が、システムの場所を問わず1on1チャットと
同等に保証される。

## 「Assistant」問題 — 命名規則

### 絶対に使わないこと
- システムプロンプト内でキャラクターを `Assistant` と呼称する
- コードコメントや変数名に `assistant_response` `assistant_name` など

### 代替表現
| NG | OK |
|----|----|
| `assistant_name` | `character_name` |
| `assistant_message` | `character_response` / `character_turn` |
| システムプロンプト内「あなたはAssistantです」 | 記述しない／キャラクター名で呼ぶ |
| ログ表示 `[Assistant]` | `[{character_name}]` で動的に置換 |

### API仕様上の例外（許容）
`role: "assistant"` はOpenAI/Anthropic APIの仕様。ここのみ許容。
ただしアプリ内部での変数名・表示名はcharacterで統一する。

## 記憶の重要性次元

`contextual` / `semantic_knowledge` / `identity` / `user_info`
評価軸は常にキャラクターの主観。ユーザ関係保全のための記憶スコアは存在しない。

## LanceDB 運用ルール

ベクトルストアは LanceDB（`data/lancedb`）に統一されている。
Lance フォーマット（追記＋アトミックコミット）により、書き込み中のプロセス停止でも
インデックスが破損しない構造的堅牢性が特徴。

### 設計概要

- **テーブル構成（単一テーブル方式）**:
  - `inscribed_memories`     — 保存記憶
  - `chat_turns`             — チャット履歴
  - `definitions`            — キャラクター定義
  - `working_memory_threads` — ワーキングメモリスレッドの index 用ベクトル
- 単一テーブル + `character_id` カラムでフィルタする方式。
  キャラクター数の増加でテーブル数が爆発しない。
- 書き込みは `merge_insert` で原子的に upsert される。

### 運用ルール

1. **新たに `LanceStore(...)` を直接生成する独立スクリプトを書く前に、
   本当に独立スクリプトが必要か再検討すること。**
   - 一括登録・バックフィル・整合性チェック等は
     backend に HTTP API（`/api/mcp/tools/call` のような内部API）を追加して
     そちらを叩くのが望ましい（MCP プロキシと同じ思想）。
   - LanceDB は multi-process write でも破損しないが、アプリ全体の単一
     インスタンス集約という設計利点は維持する。
2. **`MCPサーバ (mcp_server.py)` は backend のプロキシであり、独立して
   `LanceStore` を生成しない。** Claude CLI が再起動した際に古い実装が
   残らないよう、変更時は CLI 再起動も併せて確認すること。
3. **embedding model 変更時は Settings UI からの再インデックスを使う。**
   `LanceStore.reindex_all(new_embedding_fn, sqlite)` が SQLite を source of truth として
   全テーブルを drop → 再 embed → insert する。直接スクリプトで触らない。

### Embedding 設定

- `infinity` プロバイダー（`http://localhost:7997`、ruri-v3-310m など）を推奨。
- `google` プロバイダー（Gemini Embedding）も対応。
- sentence-transformers 系の `default` プロバイダーは依存を増やさない方針のため非サポート。

## UI / CSS デザインシステム規約

### 原則 — ページ固有属性を増やさず、システム全体で共通化・抽象化する

バックエンドUI（`backend/templates/` ＋ `backend/static/chotgor.css`）では、
**ページ専用のスタイル属性（クラス）を新設しないこと。** 必要なスタイルは、
デザインシステムの共通アトム（`ch-*`）を**再利用する**か、足りなければ
**共通アトムとして抽象的に追加する**。

| NG | OK |
|----|----|
| ページ名を冠した独自クラス（例 `log-tool-tag` を accent タグとして再実装） | 既存の共通アトムを使う（`ch-tag ch-tag--accent`） |
| 似たボタン/バッジ/ラベルをページごとに作り直す | `ch-btn` / `ch-tag` / `ch-badge` / `ch-eyebrow` 等を共有 |
| サイズ・色のバリエを個別に直書き | 抽象的な修飾子で表現（`--xs` / `--sm` / `--accent` / `--danger` / `--sub` など） |

### 根拠
属性をページ単位で増やすと「同じ見た目のものが別名で複数存在」する状態になり、
テーマ変更・配色調整・レスポンシブ対応のたびに修正漏れと不整合が生まれる。
共通アトムに集約しておけば、1箇所の変更が全ページへ一貫して波及し、DRY が保たれる。

### 実装方針
1. 新規UIを作るときは、まず `chotgor.css` 先頭の目次（TOC）で既存アトムを探して再利用する。
2. 既存アトムで表現できないときだけ、`ch-*` 名前空間に**抽象的な修飾子**として追加する
   （ページ名ではなく役割・見た目で命名。例: `ch-tag--accent`、`ch-btn--xs`）。
3. JS が参照する構造クラス/ID（`log-row`、`accordion-item` 等）は別枠。見た目アトムとは分離し、
   構造は保持・装飾は `ch-*` へ寄せる。
4. ページ固有の `log-*` 等のアトムを見つけたら、共通アトムへ正規化する（リファクタの機会に）。

## 要件・実装の提案スタンス

機能追加・改修の依頼を受けたら、実装前に必ず関連コードを確認し、改修範囲を見積もること。

- 軽微・自明な改修（タイポ、1ファイルで完結する明白な修正など）は、提案を挟まず
  そのまま実装してよい。
- 要件を少し変えることで改修範囲を**大幅に**削減できる場合のみ、着手前に提案する：
  「要件の[箇所]を[こう変更]すれば改修範囲を絞れます（可能なら 既存○ファイル→新規△
   等を併記）。変更しますか？ それともそのまま進めますか？」
- ユーザーの確認なしに、自己判断で要件を変更・省略しないこと。

### 共通化の提案
既存機能と構造が類似する場合は、抽象化・共通化による統合を提案できる：
「[既存機能]と似た構造です。まとめると[利得]が得られますが、やっていいですか？」

ただし**早すぎる共通化を避ける**。重複が2回目までは許容し、3回目／明確な利得がある
時点で抽象化する（YAGNI）。いずれもユーザーの承認後に実装すること。
